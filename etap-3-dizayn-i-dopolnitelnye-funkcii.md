# Этап 3 из 3: дизайн, дополнительные функции и финальная сборка

Завершающий этап работы над локальным аналогом DeepL. Этапы 1 (ядро + модель) и 2 (системная интеграция: хоткеи, плавающая кнопка, автозапуск) уже выполнены и работают. На этом этапе: довести интерфейс до профессионального уровня, добавить осмысленные дополнительные функции по аналогии с DeepL, собрать финальный установщик.

## 0. Обязательное свежее исследование дизайн-инструментов

Не полагайся на устаревшие знания об этих репозиториях — изучи их актуальное содержимое прямо сейчас:

1. **https://github.com/Leonxlnx/taste-skill** — набор SKILL.md против «AI-slop» в интерфейсах. На момент постановки задачи дефолтный скилл — `design-taste-frontend` (v2, с дисками VARIANCE/MOTION/DENSITY), есть строгий вариант `gpt-taste`, узкие стилевые варианты (`minimalist-skill`, `brutalist-skill`, `soft-skill`). Установи подходящий:
   ```
   npx skills add https://github.com/Leonxlnx/taste-skill --skill "design-taste-frontend"
   ```
   Проверь CHANGELOG в репо — если на момент работы v2 нестабилен, обоснованно зафиксируй v1 (`design-taste-frontend-v1`).

2. **https://github.com/pbakaus/impeccable** — флагманский анти-slop дизайн-скилл (наследник Anthropic frontend-design), с командами `/impeccable audit`, `/impeccable polish`, `/impeccable critique`, `/impeccable distill` и встроенным детектором типовых AI-штампов (фиолетовые градиенты, карточки-в-карточках, одинаковые тени и т.п.). Установи:
   ```
   npx impeccable skills install
   ```

3. **https://github.com/voltagent/awesome-design-md** — коллекция готовых `DESIGN.md` под популярные бренды. Выбери дизайн-язык, осмысленно подходящий **локальному AI-инструменту для перевода**, а не копию DeepL. Явно посмотри на кандидатов: `ollama` (терминальный, монохромный — тематически близко к «локальная модель»), `mistral` (минимализм), `claude` (тёплый редакторский стиль). Возьми один файл за основу, **адаптируй токены под собственный неймбрендинг продукта** — не копируй название/лого бренда-источника.

Зафиксируй в начале ответа: какой вариант taste-skill используешь и какой DESIGN.md взял за основу, и почему именно он подходит продукту.

## 1. Процесс редизайна существующего UI

- Это редизайн уже работающего интерфейса (не greenfield) — используй соответствующий режим/вариант из taste-skill для редизайна существующих экранов (`redesign-skill`, если он на месте подходит), а не сценарий «с нуля».
- Пройди по каждому существующему экрану (главное окно/двухпанельный переводчик, онбординг первого запуска, настройки, плавающая кнопка/тултип, трей-меню) и приведи к выбранному дизайн-языку: типографика, цветовые токены, отступы, состояния (hover/focus/disabled), тёмная/светлая тема.
- После вёрстки/редизайна каждого экрана — прогони `/impeccable audit` и `/impeccable critique`, исправь найденные типовые AI-штампы, прежде чем считать экран готовым. Не пропускай этот шаг ради скорости.
- Плавающая кнопка и тултип перевода (этап 2) должны визуально соответствовать общему дизайн-языку, а не выглядеть как отдельный нестилизованный элемент.

## 2. Дополнительные функции (по аналогии с архитектурой DeepL)

Изучи, как устроен DeepL (публичная информация о фичах десктоп-приложения), и реализуй то, что разумно ложится в полностью локальный сценарий без облака:

- **Глоссарий пользовательских терминов** — полноценный UI (не просто заглушка из этапа 1): добавление/редактирование пар «термин → перевод», привязка к конкретной языковой паре, передача в модель через параметр terminology intervention.
- **Перевод документов** — минимум `.txt`, по возможности `.docx` (через текстовый слой), с сохранением структуры/форматирования (formatted translation — нативная фича модели).
- **Множественные варианты перевода** для выделенного слова/короткой фразы (показывать альтернативы по клику).
- **Контекстный перевод** — учитывать предыдущий переведённый абзац/буфер при переводе следующего фрагмента (нативная фича модели — contextual translation).

Если решишь, что что-то из списка не стоит делать на этом этапе (например, перевод скриншота через OCR — явно отметь это как отдельный кандидат, не делай тихо без объяснения), явно укажи это в ответе и предложи как пункт будущего roadmap, а не молча пропусти.

## 3. Финальная сборка и документация

- Собери финальный установщик (`tauri build` или аналог) — компилируемый нативный бинарник/инсталлятор для Windows.
- README с: инструкцией по сборке из исходников, описанием модульной структуры (`/core`, `/os-integration`, `/ui`, `/config`, `/shortcuts`), инструкцией «как добавить новый язык / новый движок перевода в будущем», ссылкой на выбранный DESIGN.md и используемый вариант taste-skill (для прозрачности и воспроизводимости дизайн-решений), упоминанием лицензии весов HY-MT.
- Финальный прогон: первый запуск с чистой системы (без скачанной модели) → онбординг → скачивание → перевод текста → хоткеи и плавающая кнопка из этапа 2 работают на новом, отстилизованном UI без визуальных регрессий.

## 4. Критерий готовности

Полностью собранное, визуально цельное (без AI-штампов, подтверждённое прогоном через impeccable), функционально полное приложение, готовое к использованию: установщик, README, все функции этапов 1–3 работают вместе.
