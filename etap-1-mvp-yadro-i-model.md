# Этап 1 из 3: MVP — ядро приложения и локальная модель перевода

Это первый из трёх последовательных этапов разработки локального аналога DeepL на модели Tencent HY-MT. На этом этапе **дизайн и системная интеграция (хоткеи, плавающая кнопка) не нужны** — цель: получить компилируемое рабочее приложение, в котором можно ввести текст и получить перевод от локальной модели. Красивый UI и системные фичи будут добавлены на следующих этапах отдельными промтами.

## 0. Обязательное исследование перед кодом

Изучи актуальное содержимое **https://github.com/Tencent-Hunyuan/HY-MT** (и проверь, не появилось ли уже более новое репо/семейство **Hy-MT2** — на момент постановки задачи это модели 1.8B/7B/30B-A3B с открытыми весами). Не полагайся на устаревшие знания об этом репозитории. Зафиксируй и выпиши себе:

- какие форматы весов доступны (обычные, FP8, GPTQ-Int4, **GGUF**) — нас интересует GGUF, совместимый с llama.cpp;
- точный промт-шаблон модели для перевода, указанный в README (примерно: `Translate the following segment into {target_language}, without additional explanation. {source_text}`, плюс китайский вариант) — используй его буквально, не придумывай свой;
- поддерживаемые языки (33 + 5 диалектных вариаций), нативные возможности модели: terminology intervention, contextual translation, formatted translation — эти фичи должны быть заложены в архитектуру движка перевода как параметры, а не игнорироваться;
- лицензию весов;
- реальные требования к памяти для 1.8B и 7B (квантованная 1.8B может занимать ~440 МБ–1 ГБ).

В начале ответа явно зафиксируй: какую версию модели и какой конкретный GGUF-файл/квантование берёшь за дефолт, и почему.

## 1. Технологический стек — прими решение и обоснуй

Рекомендуемый (но не обязательный) стек:
- **Rust + Tauri 2** для приложения — компилируется в единый бинарник, у Tauri готовые плагины под автозапуск/трей/буфер обмена (понадобятся на этапе 2, но архитектуру закладывай сразу с расчётом на них).
- **Локальный инференс** — бандл прекомпилированного `llama-server` (OpenAI-совместимый HTTP API) как сайдкар-процесс, который Rust-бэкенд запускает при старте и опрашивает по `localhost`. Это избавляет от необходимости тащить Python/PyTorch в дистрибутив.

Если найдёшь более простой/надёжный вариант — обоснуй и предложи, но финальный результат должен оставаться **компилируемым нативным приложением для Windows**, не Python-скриптом.

## 2. Модульная архитектура (закладывается на этом этапе, используется дальше)

```
/core            — TranslationEngine (абстракция, текущая реализация = HY-MT через llama.cpp),
                   менеджер моделей (скачивание, версии, квантования, валидация),
                   менеджер языков и шаблонов промтов
/os-integration  — ПОКА ПУСТО / заглушка. Сюда на этапе 2 лягут хоткеи, плавающая кнопка, автозапуск, трей.
/ui              — Tauri-фронтенд: двухпанельный переводчик + минимальные настройки
/config          — модель настроек, сериализация (toml/json), миграции между версиями
/shortcuts       — заглушка-интерфейс для конфигурации хоткеев (реализация — на этапе 2)
```

Важно: даже если на этом этапе хоткеи и трей не реализуются, **зарезервируй для них точки расширения** в `/config` и `/shortcuts`, чтобы этап 2 не потребовал рефакторинга ядра.

## 3. Что должно работать в конце этого этапа

### Главное окно
- Двухпанельный переводчик (слева — ввод и исходный язык, справа — результат и целевой язык), с возможностью:
  - менять языки местами одним кликом;
  - менять пропорции/раскладку панелей (drag-resize, переключение гориз./вертик.);
  - автоопределение исходного языка с возможностью переопределить вручную;
  - локальная история переводов (без облака), с простым поиском.
- Дизайн на этом этапе может быть **минимальным/дефолтным** (system fonts, базовая раскладка) — это нормально, полировкой займёмся на этапе 3. Главное — функциональность и чистая разметка под будущую стилизацию (не хардкодь цвета/отступы там, где это будет мешать применить дизайн-токены позже).

### Менеджер модели
- Экран первого запуска: выбор размера модели (1.8B по умолчанию, 7B опционально), выбор квантования, путь установки (по умолчанию — `%LOCALAPPDATA%\<AppName>\models`).
- Скачивание из источника, найденного на этапе 0, с прогресс-баром, докачкой при разрыве соединения, проверкой контрольной суммы.
- Внятное сообщение при отсутствии интернета/ошибке загрузки, без необходимости переустанавливать программу.
- Возможность сменить модель/квантование позже из настроек (минимальный экран настроек на этом этапе достаточен — полноценный UI настроек дополнится на этапе 3).

### Настройки перевода (минимальный набор на этом этапе)
- Языковая пара по умолчанию.
- Переключатель GPU/CPU (если есть CUDA — оффлоадить на GPU).
- Базовый словарь терминов (даже если UI для него простой — главное, чтобы параметр доходил до модели через terminology intervention).

## 4. Критерий готовности этапа

Приложение собирается (`tauri build` или аналог), запускается, при первом запуске скачивает модель, после этого можно ввести текст, нажать «перевести» и получить корректный перевод от локальной модели с учётом выбранной языковой пары. Сборка — нативный исполняемый файл/инсталлятор, не скрипт.

## 5. Если что-то неоднозначно

Если модель окажется недоступна в GGUF на момент работы, или потребуется другой путь инференса — остановись и спроси меня, не принимай тихое решение, которое сложно будет переделать на следующих этапах.

**После завершения этого этапа — остановись и доложи результат. Этап 2 (системная интеграция: хоткеи, плавающая кнопка, автозапуск) будет отдельным промтом.**
